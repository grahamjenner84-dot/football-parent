## What this is

Two things in one repo:

1. **footballparent.co.uk** — a Next.js (App Router) SEO/content site of guides for UK football parents (academy pathways, trials, development, gear, girls' football, parent guides). Content-driven: articles are MDX files, not hardcoded JSX.
2. **Coach App** (`public/coach-app/`) — a standalone static PWA (single `index.html` + `sw.js` + manifest, no build step of its own) served at `/coach-app/app`, wired into the Next app only via `next.config.ts` redirects/rewrites/headers. It has its own service worker precache list that must be kept in sync with its own file paths — see recent commit history when touching it.

## Commands

```bash
npm run dev      # start dev server (localhost:3000)
npm run build    # production build — the real correctness check pre-commit (see below)
npm run start    # run the production build
npm run lint      # eslint (eslint-config-next core-web-vitals + typescript)
```

There is no test suite/framework configured. **`npm run build` is the primary verification step** — it runs the TypeScript check and statically generates every route, which is how broken frontmatter, bad MDX, or mismatched `getArticleBySlug()` params get caught (all article pages are static/SSG).

## Architecture: the content system

Full docs already in the repo — read these before adding/editing articles rather than re-deriving the pattern:
- `CONTENT_SYSTEM.md` — frontmatter schema, utilities, styling notes
- `QUICK_START.md` — minimal step-by-step example
- `ARTICLE_TEMPLATE.mdx` — blank template to copy

The short version:

- Each article is `content/<category>/<slug>.mdx` — YAML frontmatter (title, description, category, categoryUrl, readTime, `sections` for the sticky TOC, `relatedArticles`) + plain markdown body.
- Each article has a matching thin page at `app/<category>/<slug>/page.tsx` that calls `getArticleBySlug(category, slug)` (`lib/content.ts`), sets `metadata` (via `generateSEO()` in `lib/seo.ts`), and renders `<ArticleLayout {...frontmatter}><MDXContent content={article.content} /></ArticleLayout>`.
- **File names must match exactly** across three places: `content/<category>/<slug>.mdx`, `app/<category>/<slug>/page.tsx`, and the `getArticleBySlug("<category>", "<slug>")` call — a mismatch fails silently at build/render time, not at edit time.
- `sections[].id`/title in frontmatter drives the sidebar TOC and must correspond to `##` headings in the MDX body (TOC links are generated by slugifying heading text, not by the frontmatter `id` — check `lib/ArticleLayout.tsx` if changing heading text).
- All visual styling lives in exactly two shared components: `lib/ArticleLayout.tsx` (breadcrumbs, header, TOC sidebar, author box, JSON-LD schema for BlogPosting/Breadcrumb) and `app/components/mdx` (markdown → styled elements, e.g. callout blockquotes, tables). Change styling there, once, rather than per-article.
- Category index pages (`app/<category>/page.tsx`) are hand-written, not auto-generated from `content/`, and use the shared `app/components/category-page.tsx`.
- **New/changed routes must also be added to `app/sitemap.ts`** (a manually maintained flat route list) — it is not derived from the filesystem.
- Structured data (Organization/WebSite/Person JSON-LD) lives in `app/layout.tsx`; per-article BlogPosting/Breadcrumb JSON-LD is generated in `lib/ArticleLayout.tsx`.
- Legacy/renamed article URLs are handled via permanent redirects in `next.config.ts` — add an entry there instead of leaving a dead link when a slug changes.

## Other things worth knowing

- `internal-link-audit.mjs` is a standalone Node script (uses `app/sitemap.ts` as the source of truth for real URLs) that audits internal linking across `content/` — under/over-linked pages, weak anchor text, reciprocal-link opportunities. Run with `node internal-link-audit.mjs`; not part of the build.
- Google Analytics (gtag) is wired directly into `app/layout.tsx` via `next/script` — no analytics abstraction to route through.
- The author/site identity (name, bio, canonical site URL `https://www.footballparent.co.uk`) is hardcoded in `lib/seo.ts` and `lib/ArticleLayout.tsx` rather than pulled from config — update both if it changes.

## SEO admin page and change guardrails

- `/admin/seo` is a password-protected page (`app/admin/seo/page.tsx`, guarded by `proxy.ts` + `app/api/admin-login/route.ts`) that pulls live Search Console data via `lib/gsc.ts` and shows pages gone quiet, striking-distance keywords, low-CTR pages, decaying pages, query cannibalisation, and a rank tracker (per-query position today vs 7 days ago, each smoothed over a 3-day window since GSC's per-query position is too sample-noisy for a literal single-day comparison — real rank-tracking tools like SEMrush avoid this by running their own live daily SERP checks instead of deriving position from GSC's sampled impression data). Needs `GOOGLE_SERVICE_ACCOUNT_EMAIL`, `GOOGLE_SERVICE_ACCOUNT_PRIVATE_KEY`, `GSC_SITE_URL`, `ADMIN_PASSWORD`, `ADMIN_SESSION_SECRET` in `.env.local` (and in Vercel for production). Note: `proxy.ts` is the current Next.js convention (renamed from `middleware.ts` in v16) — this project is on a Next.js version ahead of most training data, see `AGENTS.md`.
- `scripts/generate-seo-opportunities.mjs` runs the same analysis locally and writes `seo-opportunities.json`/`seo-opportunities.md`, matching each flagged GSC page to its actual `app/<category>/<slug>/page.tsx` and `content/<category>/<slug>.mdx` files so the opportunities file is directly actionable.
- `scripts/inspect-page.mjs <path>` pulls full GSC history (monthly + weekly trend, top queries, 28-vs-28-day summary) for one specific page, for answering "why is this page doing X" questions with real data in a Claude Code session rather than guessing from the aggregate dashboard.
- **When acting on `seo-opportunities.md`/`seo-opportunities.json`, follow these rules without being reminded:**
  1. One lever per page per change (title, one section, one internal link — not a combined rewrite), so it's obvious what caused any metric movement.
  2. Commit before editing, and keep the SEO change in its own commit, never bundled with something unrelated — a bad result should be a single `git revert` away.
  3. Never change the URL/slug of a page that's already indexed and getting impressions or clicks, unless that's explicitly the point of the change.
  4. Never remove existing headings, internal links, or content sections as a side effect — additive and targeted only, no incidental restructuring.
  5. Risk-scale caution to current traffic: `striking_distance`/`cannibalisation` entries usually have little current traffic (more latitude); `low_ctr`/`decay` entries already have real clicks and rankings (be conservative, call out that it's live traffic before changing it).
  6. After publishing, that page goes on a 10-14 day watch list before touching it again — don't stack a second change before knowing whether the first worked.
  7. If a past change caused a ranking/click drop and was reverted, don't reintroduce it later even reframed, unless explicitly asked again with a stated reason.
  8. When suggesting a new title or meta description, always show the current one first (the opportunities file includes it where found) and present alternatives alongside it rather than just handing over a replacement — if the current title/meta wasn't found automatically, read the actual file before suggesting anything, don't guess at what it currently says.
  9. If a low-CTR entry is marked `resolved_by_recent_ranking_improvement` or `insufficient_recent_data`, don't treat it as an active problem — the script already re-checks aggregate low-CTR flags against a recent-only window; only entries marked `confirmed` reflect a real, current gap. The other two exist for visibility, not action.
  10. A `silence` entry (a page with real historical impressions gone near-zero recently) is not a content problem by default. Investigate first: check Search Console's URL Inspection / Test Live URL for that page, check for a recent deploy or `proxy.ts` change, check robots/noindex/canonical tags. Only consider a content edit if the technical checks come back clean — a full stop in impressions is essentially never fixed by better content when the page isn't being served or indexed properly in the first place.

  This exists because a past SEO edit cost a page its rankings entirely; reverting recovered them. The point is traceability and reversibility, not timidity.

## Editorial rules
- Never use em dashes (—) — use commas, colons, or restructure the sentence
- Never use "badge" framing clichés (e.g. "without the badge") — reads as AI filler
- Internal link suggestions must be paste-ready MDX: full surrounding sentence with [Anchor Text](/url) embedded, not bare links
